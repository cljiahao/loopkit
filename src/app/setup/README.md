# setup

## Purpose

Vendor onboarding and program-management flow at `/setup` ‚ÄĒ create, edit, migrate (change type), prep a replacement, activate, or schedule retirement for loyalty programs, with a live animated preview of the customer card.

## Contents

- `actions.ts` ‚ÄĒ server actions `saveProgramAction` (create/edit, enforces the free/Pro program-count gate), `changeTypeAction` (retire-and-replace flow for switching a program's type), `prepProgramAction` (free-tier: create a hidden second program), `activateProgramAction` (flips a prepped program to active), and `scheduleRetirementAction` (Pro-only: schedules a future cutover date to a successor program).
- `card-type-picker.test.ts` ‚ÄĒ unit tests for `card-type-picker.ts`: the 4-family/per-family-style-count shape, `isSingleStyleFamily` (true only for Stamp Card and Points Club), `resolveFamilyAndStyle`'s type/variant ‚Üí family/style mapping for every existing combination, and a round-trip check that every style's `styleToTypeAndVariant()` resolves back to itself via `resolveFamilyAndStyle`.
- `card-type-picker.ts` ‚ÄĒ pure family/style data and mapping for the type picker; groups the backend's 5 program types (2 of which already fan out into variants) into 4 vendor-facing families (Stamp Card, Growth, Points Club, Chance Card) each with their own style tiles, via `FAMILIES`, `familyOf()`, `isSingleStyleFamily()`, `resolveFamilyAndStyle()` (saved type/variant ‚Üí family/style, for the picker's active-tile highlight), and `styleToTypeAndVariant()` (style ‚Üí the type/variant pair actually saved) ‚ÄĒ no new DB type/variant value is introduced, "chance" is a UI-only grouping label.
- `page.tsx` ‚ÄĒ `SetupPage` server component; requires a vendor, applies due scheduled cutovers, seeds/reads the shared `merqo.vendor_profile` row (degrading to null on failure), resolves `resolveSetupView`'s single active view from the request's query params, and renders exactly that one view: the "Your programs" management list (`?manage=1`, with a "+ New program" link back to the plain create view), `SetupForm` (create/edit/migrate/prep), `ScheduleRetirementForm` (scheduling), or a Pro-upsell `ProLock`. A "Manage your programs" link surfaces on the plain create/upsell view whenever any programs exist.
- `preview-animation.dom.test.tsx` ‚ÄĒ jsdom test covering `usePreviewAnimation`'s ticking, celebrate-then-reset, head-start-seeded reset, immediate restart on recipe change, real win/loss rolls for lucky/wheel/scratch, and the reduced-motion static fallback.
- `preview-animation.ts` ‚ÄĒ `usePreviewAnimation()` hook; drives the real `applyVisit()`/`getProgress()` engine on a 2s timer to simulate a customer visiting, so the setup preview animates through the actual engine transitions (respects `prefers-reduced-motion`); for wheel/scratch ticks, holds the rolled result back for a `REVEAL_MS` (1400ms) `revealing` sub-phase (masking `landedId` to `null`) so the preview can play a spin/scratch anticipation animation before the win/lose signal commits ‚ÄĒ presentation-only, the real customer card has no equivalent delay since the roll already happened server-side at scan time.
- `preview-card.dom.test.tsx` ‚ÄĒ jsdom test asserting `PreviewCard` renders each view kind (dots/plant/cup/flame/chance-wheel/chance-scratch/points), name/reward fallback placeholders, win/lose popups, and the card-burst celebration overlay.
- `preview-card.tsx` ‚ÄĒ `PreviewCard` client component; a static-height snapshot of the customer card view (mirrors `ProgramCardStatus`'s view-kind switch) with an optional celebration burst and a transient win/lose popup for chance types; accepts a new `revealing` prop forwarded from `usePreviewAnimation`, passed to `Wheel` as `spinning` and to `ScratchCard` as `scratching` during the reveal-delay window.
- `preview-state.ts` ‚ÄĒ exports `buildPreviewProgram()`, `buildInitialCard()` (mirrors `enroll_card`'s head-start seed math), and `buildPreviewProgress()`; assembles a synthetic program/card from the form's current field values and runs them through the real `getProgress()`.
- `schedule-retirement-form.dom.test.tsx` ‚ÄĒ jsdom test asserting `ScheduleRetirementForm` renders a successor picker (defaulting to the first successor) and date input, and submits the program id, chosen successor, and date.
- `schedule-retirement-form.tsx` ‚ÄĒ `ScheduleRetirementForm` client component; a successor `<Select>` plus a date input, submitted via `useActionState(scheduleRetirementAction)`.
- `setup-form.dom.test.tsx` ‚ÄĒ jsdom test covering the live preview updating on keystroke/head-start toggle, submitted field values, the two-step family/style type picker (Step 1's 4 family tiles, a multi-style family expanding to its styles plus a "‚Üź Back" link, Back returning to Step 1, single-style families (Stamp Card, Points Club) completing immediately with no Step 2, and Lucky Tap now part of Chance Card's multi-style style sub-step), type-switch resetting name/reward, per-variant (Flame/Points/Cup/Sprout) type/variant/label submission behavior, and Spin the Wheel's segment editor plus its odds-weight `InfoTooltip`.
- `setup-form.tsx` ‚ÄĒ `SetupForm` client component; the full create/edit/migrate/prep form (a two-step family/style type picker ‚ÄĒ 4 family tiles, then a style sub-step for multi-style families, with a "‚Üź Back" link back to the family tiles ‚ÄĒ basics, chance-segment editor, head-start and carry-over toggles, card expiry, and ‚ÄĒ for Stamp/Plant ‚ÄĒ a separate optional `reward_expiry_days` field for reward-claim expiry) driving both submission (via `useActionState`) and the live `PreviewCard`/`usePreviewAnimation` preview. The type picker, Basics, and Rules sections are now `Section`/`ElevatedCard`-styled cards flowing in one main column; on `lg+` viewports the preview docks in a sticky side column (`lg:sticky lg:top-6`) that stays visible while scrolling through Rules, while below that breakpoint it renders inline right after the type picker (the preview is rendered twice ‚ÄĒ once per position ‚ÄĒ rather than repositioned via CSS alone, since sticky only makes sense once it's in its own grid column). Field-level microcopy stays to one short line; longer rationale/edge-case explanations (head-start's completion-lift claim, the odds-weight meaning, how card-expiry vs. reward-expiry differ) moved into `@/components/info-tooltip`'s tap-to-open `(i)` popovers instead of a native `title` attribute or a second paragraph, so Basics/Rules stay short without losing the detail. Chance Card segment editor shows a live win-percentage per segment (‚ČąNN%) next to each weight input plus an overall win-chance line above the list, computed via `program-config.ts`'s `segmentWinPercent`/`overallWinPercent` helpers.
- `setup-page.dom.test.tsx` ‚ÄĒ jsdom test calling `SetupPage` directly (mocking `listPrograms`/`isPro`/`SetupForm`/`ScheduleRetirementForm`/`getVendorProfile`) covering all 8 `SetupView` outcomes: create (Pro, with a Manage link), upsell (free-tier at cap, still with a Manage link), first-run create, the manage list (with a + New program link), edit, prep, schedule (the Pro-vendor regression case), and migrate ‚ÄĒ asserting each view is the only one rendered.
- `setup-view.test.ts` ‚ÄĒ unit tests for `resolveSetupView`: one case per `SetupView` outcome plus a regression guard for the Pro-vendor bug where an unconditionally-true `canCreate` made the `schedule` query param unreachable.
- `setup-view.ts` ‚ÄĒ `resolveSetupView()`: pure precedence logic deciding which single view `page.tsx` renders (migrate/edit/prep/schedule/manage always win over the ambient `canCreate` default, which only then picks between create/upsell) ‚ÄĒ extracted for fast, unmocked coverage, mirroring `src/app/dashboard/dashboard-view.ts`'s `shouldShowQr`.
